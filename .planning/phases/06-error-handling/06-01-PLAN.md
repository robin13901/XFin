# Plan 06-01: DAO Validation & Localized Error Messages

## Goal
Add precondition validation to critical DAO write operations, replace hardcoded error strings with localized messages, and create a custom exception class. Ensure UI forms surface DAO errors to users.

## Context
- **2 throw statements** exist in all DAOs:
  - `trades_dao.dart:167`: `throw Exception('Not enough shares to process this sell.')` — **hardcoded English**
  - `assets_on_accounts_dao.dart:246`: `throw Exception(l10n.actionCancelledDueToDataInconsistency)` — localized but generic Exception
- DAOs already receive `AppLocalizations l10n` but never use it for validation
- Localized strings already exist but are unused: `insufficientBalance`, `insufficientShares`
- `trade_form.dart` already has try/catch + `showErrorDialog(context, e.toString())` — works well
- `booking_form.dart` and `transfer_form.dart` lack try/catch around save operations
- `transfer_form.dart` already validates same-account/insufficient-shares/account-type at UI level (lines 116-146) — DAO validation serves as defense-in-depth for programmatic callers
- FK constraints enforce entity existence — no need for redundant existence checks in DAOs
- `leadsToInconsistentBalanceHistory()` already detects negative balance scenarios via `recalculateSubsequentEvents`

## New Localization Strings
Add to both `app_en.arb` and `app_de.arb`:

| Key | English | German |
|-----|---------|--------|
| `transferSameAccount` | Sending and receiving accounts cannot be the same | Sende- und Empfangskonto dürfen nicht identisch sein |
| `sharesRequired` | Shares must not be zero | Die Anteile dürfen nicht null sein |

## Tasks

### Task 1: Foundation — Exception class + localization strings + gen-l10n

**1a.** Create `lib/database/dao_exception.dart`:
```dart
/// Exception thrown when a DAO operation fails precondition validation.
class DaoValidationException implements Exception {
  final String message;
  const DaoValidationException(this.message);

  @override
  String toString() => message;
}
```
**Why `toString() => message`**: The UI calls `e.toString()` in catch blocks. Returning just the message avoids the ugly `Exception: ...` prefix that `Exception(msg).toString()` produces.

**1b.** Add to `lib/l10n/app_en.arb`:
```json
"transferSameAccount": "Sending and receiving accounts cannot be the same",
"sharesRequired": "Shares must not be zero"
```

**1c.** Add to `lib/l10n/app_de.arb`:
```json

"sharesRequired": "Die Anteile dürfen nicht null sein"
```

**1d.** Run `flutter gen-l10n` to regenerate.

### Task 2: Add validation to all four DAOs

**2a. `trades_dao.dart`**:
- Import `dao_exception.dart`
- Add `AppLocalizations l10n` parameter to `applyTradeToDb()` — update the 2 internal callers (`insertTrade`, `updateTrade`) which already have `l10n`
- Replace line 167: `throw Exception('Not enough shares to process this sell.')` → `throw DaoValidationException(l10n.insufficientShares)`
- Add at start of `insertTrade()` (inside transaction, before FIFO build):
  ```dart
  if (t.shares.value <= 0) {
    throw DaoValidationException(l10n.sharesRequired);
  }
  ```

**2b. `transfers_dao.dart`**:
- Import `dao_exception.dart`
- Add at start of `createTransfer()` and `updateTransfer()` (inside transaction):
  ```dart
  if (t.sendingAccountId.value == t.receivingAccountId.value) {
    throw DaoValidationException(l10n.transferSameAccount);
  }
  if (t.shares.value <= 0) {
    throw DaoValidationException(l10n.sharesRequired);
  }
  ```
  Note: UI `transfer_form.dart` already validates these at lines 116-128. DAO validation is defense-in-depth for programmatic callers.

**2c. `bookings_dao.dart`**:
- Import `dao_exception.dart`
- Add at start of `createBooking()` and `updateBooking()` (inside transaction):
  ```dart
  if (b.shares.value == 0) {
    throw DaoValidationException(l10n.sharesRequired);
  }
  ```
  Note: Bookings CAN have negative shares (withdrawals), so only validate non-zero.

**2d. `assets_on_accounts_dao.dart`**:
- Import `dao_exception.dart`
- Replace line 246: `throw Exception(l10n.actionCancelledDueToDataInconsistency)` → `throw DaoValidationException(l10n.actionCancelledDueToDataInconsistency)`
  This ensures consistent exception types across all DAOs and clean error display in the UI (no "Exception:" prefix).

### Task 3: Add try/catch to booking and transfer forms

**3a. `lib/widgets/forms/booking_form.dart`** (lines 313-317):
```dart
// Before:
original != null
    ? await _db.bookingsDao.updateBooking(original, companion, _l10n)
    : await _db.bookingsDao.createBooking(companion, _l10n);
if (mounted) Navigator.of(context).pop();

// After:
try {
  original != null
      ? await _db.bookingsDao.updateBooking(original, companion, _l10n)
      : await _db.bookingsDao.createBooking(companion, _l10n);
  if (mounted) Navigator.of(context).pop();
} catch (e) {
  if (mounted) showErrorDialog(context, e.toString());
}
```

## Scope Boundaries
**In scope**: Business precondition validation that the DB can't enforce; consistent exception types; UI error surfacing.
**Out of scope**:
- Entity existence checks (handled by FK constraints + UI dropdowns)
- Negative balance prevention for bookings/transfers (already handled by `leadsToInconsistentBalanceHistory()`)
- Periodic booking/transfer validation (uses silent-skip pattern; changing to throws would break batch execution)
- Account/asset CRUD validation (simple operations with UI-enforced constraints)

## Risk Assessment
- **Medium risk**: Changing `applyTradeToDb()` signature requires updating callers + tests
- **Low risk**: New validation throws — transactions will roll back automatically
- **Low risk**: Adding try/catch to forms — follows existing `trade_form.dart` pattern
